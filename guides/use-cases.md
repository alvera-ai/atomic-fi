# AtomicFi Use Cases

AtomicFi is built **compliance-first, on a speed baseline**. The premise: if regulation and latency are solved as foundational invariants — rather than reconciled after the fact — every subsequent business problem (treasury, card issuing, embedded finance, agent-driven workflows) reduces to a layer above a known-good substrate. Sections §1 through §5 enumerate that substrate: the AML, sanctions, fraud, audit, and disposition controls atomically welded into the ledger commit. Future, non-baseline use cases will be added as §6 onward; they extend the baseline rather than replacing it.

The two halves of the baseline are proven by two peer artefacts:

| Pillar           | Proof artefact                                                                              | Status      |
|------------------|---------------------------------------------------------------------------------------------|-------------|
| **Compliance-first** | This catalog (`guides/use-cases.md`) — every control as a runnable end-to-end scenario   | Live (this doc) |
| **Speed baseline**   | YCSB-style payment-flow benchmark suite (`guides/benchmarks.md`) — quantified latency / throughput envelope per workload, replayed against the same synthetic corpus this catalog uses | Forthcoming, [#25 Block 2](https://github.com/alvera-ai/atomic-fi/issues/25) |

The two artefacts share inputs by design: the synthetic compliance corpus generated by `mix alvera.gen.compliance_corpus` for the use-case tests is the same corpus the YCSB-style harness replays under load. A scenario the catalog says PASSES at 99th-percentile correctness must also pass within the latency budget the benchmark publishes.

Each scenario the platform recognises is enumerated below, organised by the regulatory regime that compels the control. Every scenario is exercised by a dedicated end-to-end test under `integration-tests/tests/scenarios/`; the **Test** column links to the file. The one-scenario-per-file invariant is deliberate: a regulator or counsel can request "scenario 23" and get exactly one piece of executable evidence with one assertion path.

---

## Result vocabulary

- **PASS** — transaction proceeds; ledger commits; no compliance event opened
- **REVIEW** — transaction held in `pending`; case opened for human review
- **BLOCK** — transaction rejected (`status="rejected"`); case opened
- **FREEZE** — holder suspended; **all** pending transactions for the holder held; high-priority case
- **OFAC report** — additionally, OFAC blocked-property report due within 10 days (31 CFR §501.603)
- **SAR-eligible** — additionally, FinCEN SAR filing due within 30 days (31 CFR §1020.320)

### 1.2 OFAC — SDN & comprehensive sanctions

Implements 31 CFR §501.404 (prohibition on dealings with blocked persons), §501.603 (blocked-property reporting), and the comprehensive country programs at Parts 560 (Iran), 587 (Russia/Ukraine), 598 (Syria), and Executive Order 13466 (North Korea).

**Two-layer screening pipeline.** Every screened party (CP, AH legal entity, beneficial owner) is checked first against the tenant's **internal blocklist** (`BlocklistMatch`, see §1.7) — any hit is fail-fast and Watchman is not queried. If no internal hit, the party is sent to **Watchman**, which screens against the OFAC SDN List, the EU consolidated list, the UN Consolidated List, and any custom watchlists provisioned at `config.all-lists.yml`. Each Watchman hit is materialised as a `SanctionsMatch` row with the underlying `match_score` (0–100), `source_list` (e.g. `"OFAC_SDN"`), `source_id` (the entity's OFAC identifier), and embedded sub-objects (`addresses`, `business_data`, `person_data`, `contact_data`). The resulting screening verdict is computed across the full set of matches.

**Graduated confidence on fuzzy matches.** OFAC's screening guidance and its 2019 *Framework for OFAC Compliance Commitments* contemplate that name-screening tools return a confidence score, not a binary verdict, and that the platform's response should be calibrated to the score. AtomicFi consumes `SanctionsMatch.match_score` and bands its response per scenarios #11–#11d below. The same graduated logic applies wherever an OFAC name-screen runs — recipient (#11), sender legal entity (#15), and beneficial owner (#29).

**OFAC 50% Rule.** Per OFAC's *Revised Guidance on Entities Owned by Blocked Persons* (August 13, 2014), an entity that is 50%-or-greater owned (in the aggregate, directly or indirectly) by one or more blocked persons is itself considered blocked even if not separately listed. Watchman returns these as matches with `sanctions_match_type = "ubo"` traversing the ownership graph; AtomicFi treats them identically to a direct SDN match. Scenario #11e exercises this path.

### 1.7 Internal tenant blocklist (`BlocklistMatch`)

Each tenant maintains its own private blocklist managed via `POST /api/blocklist-entries` (and refreshed in-memory through `BlocklistCache` + `BlocklistValidator`). Internal-blocklist screening is **fail-fast** and runs **before** Watchman: any internal hit immediately produces a `BlocklistMatch` row, marks the screening with the appropriate verdict, and skips the Watchman call entirely. This lets a tenant block a known-bad name (chargeback artist, internally-flagged fraudster, sanctions-adjacent associate, regulator-named person) without waiting for that name to land on a public OFAC list.

A `BlocklistMatch` row carries `matched_term`, `match_type` (`exact` | `regex`), `scope` (`first_name` | `last_name` | `company_name`), `reason` (free-text rationale recorded by the blocklisting reviewer), and `blocklist_updated_at` (the list's last-refresh timestamp at screening time). The `false_positive_qualifier` field persists reviewer dispositions across re-screenings — see §4.

### 1.8 PEP & adverse media (FATF R12, FFIEC EDD)

Implements FATF Recommendation 12 (Politically Exposed Persons), FinCEN's PEP guidance, and the FFIEC BSA/AML Examination Manual chapter on PEPs and adverse media. PEP and adverse-media checks are **distinct screening passes** from OFAC sanctions: AtomicFi's `ComplianceScreening.screening_type` enum carries dedicated `:pep` and `:adverse_media` values, and the screening worker dispatches each pass independently. Watchman returns PEP designations against the U.S. Consolidated Sanctions List (CSL), the EU PEP list, and any tenant-extended PEP list shipped via `custom-watchlist.jsonl`.

A PEP is not automatically a blocked person — PEP status mandates **enhanced due diligence**, periodic re-screening, and a heightened-monitoring overlay; it does not by itself prohibit dealings. AtomicFi treats every PEP-positive holder or counterparty as REVIEW on origination above a configurable amount and re-screens periodically (6 months for foreign PEPs, 12 months for domestic). Adverse media (negative-news hits) is treated symmetrically.

### 1.9 Custom tenant-extended watchlist (Watchman Senzing JSONL)

Beyond OFAC SDN, Consolidated Sanctions List, FinCEN §311 advisories, and UN CSL — all of which Watchman ingests by default per `config.all-lists.yml` — each tenant may extend Watchman's screening pass with its own custom list. The list lives at `custom-watchlist.jsonl` in Senzing JSONL format and is refreshed by Watchman on the configured interval (12h by default). Custom-watchlist entries support both `RECORD_TYPE: "PERSON"` (with `NAME_FIRST`, `NAME_LAST`, `DATE_OF_BIRTH`, `PASSPORT_NUMBER`, etc.) and `RECORD_TYPE: "ORGANIZATION"` (with `NAME_ORG`, `TAX_ID_NUMBER`, etc.).

Distinct from §1.7 internal blocklist: §1.7 is **fail-fast and pre-Watchman** (any internal hit short-circuits Watchman entirely); §1.9 is **fused into Watchman's screening pass** and produces a `SanctionsMatch` row with `source_list = "CUSTOM_WATCHLIST"` and `source_id = <RECORD_ID>`. Both honour the FP-disposition workflow defined in §4.

### 1.10 Fail-closed posture & screening availability

AtomicFi's compliance posture is **fail-closed**: when a screening provider (Watchman) is unreachable, the in-memory blocklist cache is stale beyond its tolerance, or the screening worker exhausts its retry budget, the platform refuses to settle dependent transactions rather than letting them through unscreened. This is the only posture defensible against an OFAC examiner who asks "what would have happened if Watchman was down?"

### 1.11 Continuing-SAR monitoring (FinCEN 2000-002)

After a Suspicious Activity Report is filed (§4 D5), FinCEN's *Suspicious Activity Reporting Guidance* (October 2000, FIN-2000-002) directs heightened monitoring of the same subject. AtomicFi marks the AH with a continuing-SAR flag for **90 days** following the SAR filing; subsequent transactions are placed in REVIEW (rather than passing) regardless of whether they would otherwise have triggered a rule. After the 90-day window closes with no further suspicious activity, the flag clears and the AH returns to baseline monitoring.

## 4. Reviewer disposition & evidence storage

Every REVIEW outcome above lands in the case-management UI for a human reviewer (compliance officer) to disposition. This section enumerates the operational paths a case can take from `open` to `closed`, the evidence the reviewer captures along the way, and the storage backing for that evidence.

**Document storage architecture.** All uploaded documents — KYC IDs, proof-of-address scans, beneficial-owner identification, lawful-order PDFs, reviewer disposition justifications, SAR-pack attachments — flow through `POST /api/documents` and land in object storage:

- **Development / demo** (`make up`): **LocalStack S3**, container provisioned by `local-dependencies.yaml`, bucket `atomicfi-evidence-local`, no encryption, no Object Lock, ephemeral. Sufficient for offline demos and test runs.
- **Staging / production**: **AWS S3** (or equivalent regional provider), bucket `atomicfi-evidence-{env}-{tenant_id}` with: SSE-KMS encryption (CMK per tenant), versioning enabled, S3 Object Lock in `compliance` retention mode for documents tagged `legal-hold` (lawful-order PDFs, SAR-pack attachments), 5-year minimum retention to satisfy 31 CFR §1010.430, server-access logging to a separate audit bucket, and IAM access scoped to the document's tenant.

The `documents` table holds metadata only (filename, content-type, size, sha-256, S3 key, classification, linked subject); the binary body is never in Postgres. Every read of a document writes an `audit_events` row (`event="document_accessed"`).

**Disposition vocabulary.**

- **FALSE POSITIVE (FP)** — reviewer determines the rule fired in error (e.g. an SDN name match was a different person). Case closes; transaction proceeds; reviewer's justification text and any supporting document are recorded for OFAC/FFIEC audit trail. Mechanically, the platform writes `false_positive_qualifier = "manual_override"` to the underlying `SanctionsMatch` (or `BlocklistMatch`) row, with `reviewed_by_user_id`, `reviewed_at`, and `review_notes`.
- **TRUE POSITIVE (TP)** — reviewer confirms the rule fired correctly. Where the original verdict was REVIEW, it escalates to BLOCK and (where applicable) the OFAC blocked-property report or SAR-eligible tag is set.
- **REQUEST DOCUMENTS** — reviewer pauses the case awaiting additional evidence from the AH; case state transitions to `awaiting_documents`; transaction stays in `pending`. AH is notified through whatever channel the tenant configures.
- **APPROVE ON UPLOAD** — AH uploads the requested document; reviewer reviews and marks approved; case auto-resolves; transaction settles.
- **ESCALATE TO SAR PACK** — for `sar-eligible` cases, reviewer triggers assembly of the FinCEN SAR pack (transaction history + AH KYC docs + beneficial-owner docs + case timeline). Pack is downloadable for filing in BSA E-Filing.

**FP persistence across re-screenings (auto-suppression).** Because a sanctioned-name lookup may return the same false-positive match every time the same AH/CP is screened, AtomicFi automatically reapplies prior FP dispositions on subsequent screenings. Before computing the screening score, the `ComplianceScreeningContext` queries:

```sql
SELECT source_id FROM sanctions_matches
 WHERE tenant_id  = $tenant_id
   AND source_id  = $source_id
   AND false_positive_qualifier IN ('manual_override','auto_suppressed')
 LIMIT 1;
```

If a prior override exists, the new match is written with `false_positive_qualifier = "auto_suppressed"` and is **excluded from `screening_score`**. The same logic applies to internal blocklist matches keyed on `(matched_term, scope)`. This satisfies the OFAC Framework's expectation of a documented, repeatable FP-disposition workflow without forcing reviewers to re-disposition the same noise on every transaction.

**List-refresh re-evaluation.** Each match row stores `list_synced_at` (sanctions) or `blocklist_updated_at` (internal blocklist) — the source list's last-refresh timestamp at the moment the match was made. When a reviewer-set `manual_override` is older than the list's most-recent refresh, the platform surfaces the case for re-evaluation: a list update may have introduced new identifying details (date of birth, alias, address) that change the override calculus.

## Glossary

- **Account-event** — a non-financial change to the AH or its linked legal entity: address, beneficiary, email, phone, password reset, KYC re-affirmation. Recorded as a row in `audit_events` with `event="account_event_changed"`. Fraud rules in §2.1 join `transactions` against this stream within rolling time windows.
- **AH** — Account Holder (the platform's MDM subject — `account_holders` row, linked to a `legal_entities` row that carries the PII).
- **ATO (account takeover)** — third-party gains control of a legitimate AH's credentials and uses the account to move funds. Detected by §2.1 rules #23 and #26.
- **CP** — Counterparty (an external party on a transaction, customer or non-customer — `counterparties` row).
- **Dormancy reactivation** — long-inactive account suddenly becomes active. Fraud / mule signal because dormant accounts are frequently sold or compromised (#25).
- **PA** — Payment Account (the account a debit leaves from or a credit lands in — `payment_accounts` row, linked to AH for the originator side, to CP for the beneficiary side).
- **PPSI** — Permitted Payment Stablecoin Issuer (GENIUS Act §3).
- **Sub-CTR band** — transactions in the USD 9,000–9,999 range, structured to evade the USD 10,000 Currency Transaction Report threshold.
- **Travel Rule** — FATF Recommendation 16 / 31 CFR §1010.410(f), requiring originator + beneficiary identification to travel with every wire ≥ USD 3,000 (or, for stablecoins under the GENIUS NPRM, ≥ USD 1,000).
- **Unhosted wallet** — a self-custody on-chain address controlled by a private-key holder rather than a regulated PPSI or exchange.
- **Watchman score** — the 0–100 confidence score returned by the Moov Watchman screening service for a fuzzy name match against an OFAC list. AtomicFi consumes this score and bands its response per §1.2 (#11–#11d). The score is recorded on the `compliance_screenings` row regardless of verdict, so a regulator can audit the false-positive disposition trail.
- **PEP (Politically Exposed Person)** — natural person entrusted with prominent public function (head of state, senior politician, senior judicial / military / SOE official, senior international-org official) plus their immediate family and close associates. AtomicFi distinguishes **foreign PEP** (domiciled outside the home jurisdiction; FATF R12 mandatory EDD, 6-month re-screen) from **domestic PEP** (domiciled in the home jurisdiction; risk-based EDD, 12-month re-screen). Status is recorded in `legal_entities.politically_exposed_person` and screened independently via `ComplianceScreening.screening_type = "pep"`.
- **Adverse media** — negative-news hits associating a screened party with fraud allegations, regulatory enforcement, criminal investigation, or other reputational risk. Screened independently as `ComplianceScreening.screening_type = "adverse_media"`. A positive hit triggers REVIEW, never automatic BLOCK (FFIEC requires human disposition for media-driven flags).
- **Custom watchlist** — tenant-extended screening list shipped as `custom-watchlist.jsonl` in Senzing JSONL format and ingested by Watchman on the configured refresh cadence. Distinct from §1.7 internal blocklist (which is fail-fast and pre-Watchman); custom-watchlist hits flow through Watchman's normal pass and produce `SanctionsMatch` rows with `source_list = "CUSTOM_WATCHLIST"`.
- **Fail-closed** — operational posture where a transaction is held (REVIEW) rather than allowed to settle when the platform cannot complete a required compliance check. AtomicFi is fail-closed by design (§1.10).
- **Continuing-SAR window** — 90-day heightened-monitoring period following a SAR filing, per FinCEN FIN-2000-002. Subsequent transactions by the SAR-filed AH are placed in REVIEW even if they would otherwise pass; window auto-clears after 90 days of clean activity.
- **`BlocklistMatch`** — schema row representing a hit against the tenant's internal blocklist. Carries `matched_term`, `match_type`, `scope`, `reason`, `blocklist_updated_at`, and the FP-disposition fields. Internal-blocklist screening is fail-fast and runs before Watchman (§1.7).
- **`SanctionsMatch`** — schema row representing a hit returned by Watchman against an external sanctions list (OFAC SDN, EU consolidated, UN consolidated, custom). Carries `matched_name`, `match_score` (0–100), `sanctions_match_type` (`exact` | `fuzzy` | `ubo` | `entity`), `source_list`, `source_id`, embedded sub-objects (`addresses`, `business_data`, `person_data`, `contact_data`), `list_synced_at`, and FP-disposition fields.
- **`false_positive_qualifier`** — enum on both `BlocklistMatch` and `SanctionsMatch` (`none` | `manual_override` | `auto_suppressed`). Persists reviewer FP decisions across re-screenings; see §4 D1, D6.
- **OFAC 50% Rule** — OFAC's *Revised Guidance on Entities Owned by Blocked Persons* (Aug 13, 2014). An entity that is 50% or more owned in the aggregate by one or more blocked persons is itself blocked, even if not separately listed. Watchman returns these matches with `sanctions_match_type = "ubo"`. Scenario #11e.
- **FP / TP** — false positive / true positive. A reviewer's disposition of a previously-flagged case (§4 D1, D2). FP closes the case and lets the transaction proceed; TP escalates the verdict (REVIEW → BLOCK where applicable) and triggers downstream regulatory filings. Both dispositions are recorded with the reviewer ID, justification text, and any uploaded supporting evidence.
- **SAR pack** — a downloadable bundle assembled by the platform when a case is escalated for FinCEN Suspicious Activity Report filing (§4 D5). Contains the transaction history, the AH's KYC document set, beneficial-owner documents, and the case-event timeline, formatted for attachment to the BSA E-Filing submission.
- **Object Lock (S3)** — write-once-read-many (WORM) retention mode in AWS S3. Lawful-order documents and SAR-pack attachments are stored under `compliance` retention so they cannot be altered or deleted before the legal-hold period expires, satisfying SOX §802 anti-tampering and 31 CFR §1010.430 retention.
- **LocalStack** — open-source local AWS emulator used by `make up` to provide an S3-compatible bucket for evidence storage in development. Production deployments swap LocalStack for real AWS S3 (or equivalent) via environment configuration; no application code changes.
