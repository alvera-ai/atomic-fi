# Check API Quality

Structural CHECKER for a single atomic-fi controller + its test file. Runs
ripgrep-scoped checks against atomic-fi's controller / context / OpenApiSpex
conventions and prints a pass/fail list. This skill **does not fix** —
it surfaces drift; the caller decides whether to patch the controller, the
test, or the convention.

## Usage

```
/qa:check-api-quality lib/atomic_fi_api/controllers/<resource>_controller.ex
```

**Examples:**
```
/qa:check-api-quality lib/atomic_fi_api/controllers/account_holder_controller.ex
/qa:check-api-quality lib/atomic_fi_api/controllers/transaction_controller.ex
```

If no argument is given, ask for a controller path.

---

## What the skill verifies

Each check maps to an atomic-fi convention documented in
[`CLAUDE.md`](../../../CLAUDE.md) (Controller / Context Contract, OpenAPI
Schema Patterns) or the maker side at
[`/dev:create-rest-api`](../dev/create-rest-api.md).

### Controller invariants

| # | Check | How (ripgrep against the controller file) |
|---|---|---|
| C1 | `use OpenApiSpex.ControllerSpecs` present | `Grep "use OpenApiSpex.ControllerSpecs"` |
| C2 | `operation :…` block for each public action | count `^  operation :` ≥ count of public `def` actions |
| C3 | `tags [...]` set once, matches an `ApiSpec` tag | `Grep 'tags \['` — cross-ref with `lib/atomic_fi_api/api_spec.ex` |
| C4 | Body-params arrive as `%FooRequest{}` struct — never `Map.from_struct/1` or `ExOpenApiUtils.Mapper.to_map/1` in the controller (per `CLAUDE.md` § Controller / Context Contract) | `Grep "Map\\.from_struct\|Mapper\\.to_map"` must return **zero** hits |
| C5 | Responses built via `ApiHelpers.json_response/3` or `.json_paginated_response/4` — never ad-hoc `json(conn, %{data: struct})` | `Grep "json\\(conn, %\\{data:"` must return zero hits |
| C6 | Session pulled via pattern match on `:api_session` in handler head — not `conn.assigns.api_session` or `conn.assigns.current_session` | `Grep "conn\\.assigns\\.(api_session\|current_session)"` must return zero hits |
| C7 | No `PATCH` — only `PUT` for updates | `Grep "patch :"` in the operation blocks + cross-ref router |
| C8 | Side effects (Oban jobs, async work) live in the context layer, not the controller | `Grep "Oban\\.\|enqueue\\("` must return zero hits in the controller |

### Router invariants

| # | Check | How |
|---|---|---|
| R1 | Routes scoped under `scope "/api", AtomicFiApi do` (tenant-only — atomic-fi does not have datalake-scoped routes) | `Grep "scope \"/api\""` in `lib/atomic_fi_api/routes.ex` |
| R2 | Pipeline uses `AtomicFiApi.Plugs.ApiAuthentication` (NOT `fetch_api_key` or any platform-style plug) | `Grep "ApiAuthentication"` |
| R3 | `put/4` used, not `patch/4` | `Grep "^\\s+patch\\s" lib/atomic_fi_api/routes.ex` — flag any hits |

### Schema registration invariants

Per `CLAUDE.md` § OpenAPI Schema Patterns:

| # | Check | How |
|---|---|---|
| S1 | For each `%FooRequest{}` / `%FooResponse{}` aliased in the controller, the module is auto-generated by an Ecto schema's `open_api_schema(...)` macro | `Grep "open_api_schema" lib/atomic_fi/...` for the parent schema |
| S2 | Auto-generation requires `title:` matching module name exactly — no spaces (`"AccountHolder"` ✓, `"Account Holder"` ✗) | `Grep "title: \"[^\"]+ [^\"]+\"" lib/atomic_fi/...` — any hit with a space is drift |
| S3 | Tag registered in `lib/atomic_fi_api/api_spec.ex` | `Grep "<controller's tag>" lib/atomic_fi_api/api_spec.ex` |
| S4 | Server-generated fields (`id`, `inserted_at`, `updated_at`, `tenant_id`) declared with `readOnly: true` so they're excluded from Request schemas | `Grep "readOnly: true" lib/atomic_fi/...<resource>.ex` — must hit at least once per persistent field |
| S5 | Nested embedded schemas reference auto-generated Request/Response variants via `$ref`, not bare module names | `Grep '\\$ref.*FooRequest\\|\\$ref.*FooResponse'` in schema files |

### Context invariants (the controller is just the entry — the contract is enforced one layer down)

| # | Check | How |
|---|---|---|
| X1 | Context function pattern-matches the typed struct in the head: `%FooRequest{} = request` | `Grep "def.*%[A-Z][A-Za-z]+Request{}\\s*=\\s*request" lib/atomic_fi/<context>/` |
| X2 | Context passes the struct directly to `changeset/2` (the `ExOpenApiUtils.Changeset.cast` macro calls `Mapper.to_map` internally — no manual conversion needed) | `Grep "Map\\.from_struct\|Mapper\\.to_map" lib/atomic_fi/<context>/` must return zero hits |
| X3 | Context functions use `def_with_rls_and_logging` (RLS + audit log macro) for read/write operations | `Grep "def_with_rls_and_logging" lib/atomic_fi/<context>/` |

### Test invariants

| # | Check | How (ripgrep against the companion `*_controller_test.exs`) |
|---|---|---|
| T1 | `use AtomicFiWeb.ConnCase` (async pref is per-test; `async: false` is acceptable for tests that touch shared resources like Watchman) | `Grep "use AtomicFiWeb.ConnCase"` |
| T2 | `import OpenApiSpex.TestAssertions` | `Grep "import OpenApiSpex.TestAssertions"` |
| T3 | `import AtomicFi.Factory` — no hand-rolled tenant/session/api-key fixtures inline | `Grep "import AtomicFi.Factory"` |
| T4 | Every `json_response(conn, 200 \| 201)` is followed by an `assert_schema/3` call in the same test | line-pair check: for each `json_response(conn, 20[01])`, the next 10 lines contain `assert_schema(` |
| T5 | Create / update tests end with a DB cross-check via the context function | in `describe "POST create"` + `describe "PUT update"` blocks, `Grep` for a `<Context>.get_…!(session, …)` match |
| T6 | No `patch(conn, …)` in the test file (updates must use `put/3`) | `Grep "^\\s+patch\\(conn"` must return zero hits |
| T7 | Tests cover the RLS isolation invariant — at least one test scopes to tenant A and asserts a tenant-B resource is not visible | `Grep "tenant_a\\|tenant_b\\|other_tenant\\|RLS"` (heuristic; flag if zero) |
| T8 | At least one negative test: invalid request body asserts a 4xx with schema-validated error response | `Grep "json_response\\(conn, 4"` + `assert_schema` follow-up |

## Report format

Emit a single pass/fail list — one line per check, ordered by severity
(controller → router → schemas → context → tests). Example:

```
lib/atomic_fi_api/controllers/account_holder_controller.ex
──────────────────────────────────────────────────────────
✅ C1  OpenApiSpex.ControllerSpecs in use
✅ C2  operation block for every public action
❌ C4  Map.from_struct/1 in controller body_params (line 147)
       → pass %AccountHolderRequest{} straight to context; let
         ExOpenApiUtils.Changeset.cast handle conversion internally
❌ C6  conn.assigns.api_session (lines 82, 114)
       → destructure %{assigns: %{api_session: session}} in handler heads
✅ C7  no patch operations
✅ C8  no Oban / side effects in controller
✅ R1  routes are tenant-scoped under /api
✅ R2  ApiAuthentication plug wired
✅ S1  every referenced schema resolves via open_api_schema
✅ S3  tag "AccountHolders" registered in api_spec.ex
❌ S4  inserted_at/updated_at on AccountHolder missing readOnly: true
       → these will leak into AccountHolderRequest — add readOnly: true
✅ X1  context functions pattern-match %AccountHolderRequest{} in head
✅ X3  def_with_rls_and_logging used
❌ T3  hand-rolled session fixtures; AtomicFi.Factory unused at lines 34-58
❌ T4  json_response(conn, 200) without assert_schema at test:account_holder_controller_test.exs:156
❌ T7  no RLS isolation test detected
───────────────────────────────────────────────────────────
5 failures. Fix before opening PR.
```

If a check is N/A for a controller (e.g. T7 isn't required for tenant-management
endpoints that operate at the cross-tenant level), report it as `⊘ Tn  N/A:
<reason>` rather than skipping silently.

## What the skill does NOT do

- **Does not** run `mix test` or `mix credo` — that's `/qa:quality-checks`.
- **Does not** check OpenAPI spec YAML is regenerated — deferred to the
  maker flow.
- **Does not** validate authorisation logic at runtime (API key scope, role
  gates). Those are runtime concerns; this skill audits structure. For
  reach-level review, pair with `/qa:review`.
- **Does not** mutate files. It reports; you fix.
- **Does not** check ZEN-rule wiring for use-case scenarios — that's separate
  Block 1 work (see [`guides/use-cases.md`](../../../guides/use-cases.md) +
  Phase 2 of [#27](https://github.com/alvera-ai/atomic-fi/issues/27)).
