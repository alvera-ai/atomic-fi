#### A.2 — Bootstrap extension: second tenant for RLS

Every resource spec MUST have an RLS test (per skill rule §4). To assert tenant B can't see tenant A's data, you need a pre-minted second tenant.

Build:
- Extend `bootstrap.test.ts` (or add `bootstrap-secondary.test.ts` that runs after bootstrap via filename ordering — `bootstrap_b.test.ts`?) to:
  1. Read the Platform Admin API key from the DB (it's randomly generated in `priv/repo/seed_migrations/20260501000001_seed_system_entities.exs:128`). Two options:
     - **Recommended:** add a small helper that runs `psql` via `child_process.spawn` to fetch it once, OR add an env var `PLATFORM_ADMIN_API_KEY` that the user supplies.
     - Quick path: extend the seed migration to write the value to a known location (e.g. `priv/repo/.platform_admin_api_key` — gitignored) on first creation, so tests can pick it up.
  2. `POST /api/tenants` as the platform admin to create a fresh tenant scoped to this `runId` (name like `e2e-secondary-${runId}`).
  3. Mint a tenant_admin user inside the secondary, sign in, save bearer.

Persist into `vitest-state/<runId>/bootstrap.state.json`:
```ts
{
  primary:   { tenantId, tenantSlug, bearer, rootApiKey, ... },
  secondary: { tenantId, tenantSlug, bearer }
}
```
Update `BootstrapState` type and `requireBootstrap()` accordingly.

**Decision point for the user:** does the secondary tenant get torn down at end of run, or is it left for inspection? Default to leaving it (cheap; named by runId so collisions are impossible).

#### A.4 — Per-resource spec template

Drop in `integration-tests/tests/e2e/_template.test.ts.tmpl` (NOT a `.test.ts` so vitest doesn't pick it up). Every new spec is a copy-paste from this. The template covers all 10 standard cases from the skill:

1. Index — empty (or fresh tenant, secondary)
2. Create — happy path → 201, UUID
3. Create — invalid → 422 with errors
4. Show — by id → 200
5. Show — random UUID → 404
6. Update — happy path → 200
7. Update — invalid → 422
8. Delete → 204
9. Index — pagination
10. RLS isolation (secondary tenant)

Resources without a full CRUD surface (compliance-screenings is POST + GET-by-id only; sessions are special; etc.) get tailored templates — start from the matching ExUnit test under `test/atomic_fi_api/controllers/<resource>_controller_test.exs` for ground truth.

#### A.5 — Sequencer (filename prefixes)

Use-case ordering will eventually need this; resource specs mostly don't depend on each other (only on bootstrap). But a few do:
- `ledger-entries` requires `ledgers` + `ledger-accounts` to exist
- `transactions` requires `payment-accounts`
- `beneficial-owners` requires `legal-entities`
- `compliance-screenings` requires `account-holders` (or a similar parent)

Two options:
- **Filename prefixes** (`00_bootstrap`, `10_ledgers`, `11_ledger-accounts`, `12_ledger-entries`, ...). Simple, alphabetical sort already works.
- **Custom sequencer**: blocked in vitest 2.1.9 — `BaseSequencer is not a constructor`. Either upgrade vitest to 3.x or stick with prefixes.

**Recommend filename prefixes** unless the user pushes back. Cheap, transparent, debuggable.

If you adopt prefixes, rename `bootstrap.test.ts` → `00_bootstrap.test.ts` and update `_order.json` (or just delete it; the prefixes are the order).

#### A.6 — Spec regeneration check

The OpenAPI spec is committed at `packages/sdk/spec/openapi.yaml`. Any PR touching `lib/atomic_fi_api/` should regenerate it. Add a CI step (or at minimum a `pnpm sdk:check` script) that runs `mix openapi.spec.yaml --check=true` (the OpenApiSpex `--check=true` flag verifies the file matches the live module without writing). Out-of-scope to wire CI; just leave the script in `package.json` and document it.

#### A.7 — Test-isolation hygiene

Resource specs create real rows in `atomic_fi_dev`. After many runs, the DB will be cluttered. Either:
- Each spec deletes what it creates in an `afterAll` (clean but fragile if assertions fail mid-test).
- Or: each runId gets a fresh tenant (preferred — already half-done via the secondary tenant idea, just extend to a *primary-per-runId* pattern).

**Recommended:** use the seeded `atomic-fi-tenant` as a stable platform-admin tenant, but every resource spec that creates rows scopes them under a per-runId child tenant (created at bootstrap). Tear down nothing; the tenant name namespaces everything. CI would `mix ecto.reset` between runs anyway.

Discuss with user before committing to this — it changes the bootstrap shape from `{primary, secondary}` to `{platform, primary-runId, secondary-runId}` or similar.

## Known papercuts you'll hit

- **`mix.exs` version drift**: `mix.exs` is `0.1.0` but `.release-please-manifest.json` says `1.1.0`. `release-type: simple` doesn't auto-bump. SDK package version mirrors `mix.exs` via `pnpm version:sync`. Don't conflate.
- **Vitest sequencer in 2.1.9**: extending `BaseSequencer` hits `not a constructor` at runtime. We removed the custom sequencer. Use filename prefixes for ordering instead.
- **`@hey-api/openapi-ts` 0.97**: the `client` is a plugin (`'@hey-api/client-fetch'` in the `plugins` array), not a top-level `client:` field. The skill SKILL.md mentions 0.62 syntax — ignore that, the config in `packages/sdk/openapi-ts.config.ts` is current.
- **Bearer expiry**: bootstrap mints with `expires_in: 3600`. Re-running specs in the same runId will re-use the cached bearer. If a spec needs short-lived auth (e.g. asserting expiry), mint its own session.
- **Misleading seed log**: `priv/repo/seeds.exs` says "Seeded N new" even on idempotent re-runs. Counter is wrong, no actual duplicates inserted. User said they don't care.
- **Recordings**: `recordingFetch` infrastructure does NOT exist yet and is not needed for this handover (it's a use-case-only concern). Skip it. If you find yourself building it, you've crossed the line into use-cases.
